Ziel & Prinzip (in zwei Sätzen)
Der Monitor sendet Pixel per HDMI/TMDS mit dem Pixeltakt 148,5 MHz (1080p60); die ungewollte Abstrahlung liegt auf dessen Harmonischen — genau deinen 148,5 / 297 / 445,5 MHz. Du fängst diese Emanation mit dem SPECTRAN V6 als komplexes IQ ab und lässt ein DRUNet lernen, aus dem verrauschten komplexen Signal das lesbare Originalbild zu rekonstruieren — ohne AM-Demodulation, direkt aus den IQ-Samples (das ist der Kernpunkt von deep-tempest).
Vorgehen in 4 Phasen
Phase 0 — Setup (einmalig). Ubuntu-Laptop: requirements.txt installieren, dazu CUDA-Torch passend zur NVIDIA-Karte und tesseract-ocr. Warum: Die GPU trägt das Training, Tesseract liefert die OCR + die CER-Metrik, an der du den Fortschritt misst.
Phase 1 — Bootstrap auf synthetischen Daten. gen_clean_text.py erzeugt Text-Screenshots, train.py --clean trainiert darauf. Warum zuerst synthetisch: DRUNet braucht pixelgenau ausgerichtete Paare (degradiert ↔ sauber). Bei echten Captures ist genau diese Ausrichtung das schwierigste Problem. Synthetisch sind die Paare perfekt ausgerichtet und unbegrenzt verfügbar — du bringst das Netz so ohne einen einzigen Capture in die Nähe der Lösung. Wirf hier auch echte Screenshots deiner Ziel-Oberflächen (Terminals, Login-Masken) mit rein.
Phase 2 — Echte Captures + Fine-Tuning. Im Zelt: V6 auf 445,5 MHz (3. Harmonische, beste Kantenschärfe für Text), Span groß, ein paar Sekunden pro Motiv aufnehmen, dazu jeweils das Original-PNG ablegen. build_pairs.py mittelt über viele Frames (das Bild ist statisch → starke Rauschminderung) und richtet jeden Capture per Kreuzkorrelation gegen das bekannte Original aus — dein Laborvorteil. train.py --real --init justiert das Netz dann an deine reale Hardware/Antenne/Umgebung. Warum: Nur echte Captures enthalten die tatsächlichen Kanaleffekte; Fine-Tuning holt die letzten CER-Prozentpunkte.
Phase 3 — Auswertung. Zwei Wege:

Offline, beste Qualität: infer.py auf ein IQ-Capture → Bild + OCR-Text + CER gegen das Original. Zielgröße laut Paper: von ~90 % CER (roh) auf < 30 % (mit Netz).
Live über HTTP: In RTSA-Suite PRO SPECTRAN V6 → HTTP Server Block bauen, live.py zieht den Strom (fs = span × 1.5) und rekonstruiert fortlaufend nach live.png. Warum HTTP: Echtzeit-Blick zum Justieren und für Demos; ohne --model siehst du die reine gemittelte Rohrekonstruktion, ideal zum Ausrichten von Frequenz und Antenne.

Kompakte Befehlsreihenfolge
bash# Phase 0
pip install -r requirements.txt
pip install torch --index-url https://download.pytorch.org/whl/cu121

# Phase 1 — synthetisches Bootstrap
python gen_clean_text.py --out data/clean --n 400
python train.py --clean data/clean --out runs/synth --iters 40000 --batch 8

# Phase 2 — echte Captures (captures/motivNNN.cf32 + .png), dann Fine-Tuning
python build_pairs.py --in captures --out data/pairs --fs 122e6 --timing 1080p60 --preview
python train.py --real data/pairs --init runs/synth/best.pt --out runs/real --iters 8000

# Phase 3a — Offline-Auswertung
python infer.py --model runs/real/best.pt --iq neu.cf32 --fs 122e6 --harmonic 3 --gt original.png --out out.png

# Phase 3b — Live über HTTP
python live.py --url "http://<laptop-ip>:54664/stream?format=float" --fs 15e6 --harmonic 3 --model runs/real/best.pt --out live.png --ocr

Die drei Stellschrauben, wenn Ergebnisse schlecht sind: zuerst Frame-Sync/Ausrichtung prüfen (--preview in build_pairs.py), dann fs verifizieren, erst danach am Netz drehen. Fast alle Probleme liegen an Sync oder Sample-Rate, nicht am Modell.

Wenn du im Labor bist und es hakt, sind die zwei häufigsten Ursachen fast immer der Frame-Sync (Ausrichtung mit --preview prüfen) oder eine falsch gesetzte Sample-Rate fs — beides bevor du am Netz zweifelst.
Viel Erfolg beim Aufbau — sag Bescheid, wenn du die Zwei-Geräte-Fusion (297 + 445,5 MHz gleichzeitig) dazunehmen willst oder beim echten Capture etwas nicht wie erwartet aussieht.
